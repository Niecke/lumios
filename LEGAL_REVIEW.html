Part 1 — Critical: Must Fix Before Go-Live

These items represent false claims in the legal documents or missing legally required functionality. Each one is a compliance risk.
Critical Implement EXIF / GPS stripping on upload
Legal claim: AVV §2(1), Datenschutz “Fotodaten”, TOM Annex all state “Automatische Entfernung von EXIF-Metadaten (insbesondere GPS-Standortdaten) beim Upload”.
Reality: Only ImageOps.exif_transpose() is called — this reads orientation but does not strip EXIF. Original files are uploaded to GCS with full GPS coordinates intact. Previews/thumbs may lose EXIF incidentally via Pillow re-save, but this is not guaranteed.
Fix: After exif_transpose(), explicitly strip all EXIF data from the original image before uploading to GCS. Simplest approach: re-encode the original through Pillow without copying EXIF info tags.
File: backend/app/blueprints/api/images.py:209
Critical Remove Apple Sign In from Datenschutz
Legal claim: Datenschutz SSO section contains an entire “Apple Sign In” block describing data processing with Apple Inc., SCC transfer basis, and privacy policy link.
Reality: Apple Sign In is not implemented anywhere in the backend. No Apple OAuth code, no Apple client ID config, no Apple token verification. Only Google OAuth exists.
Fix: Remove the entire “Apple Sign In” <h3> block from Datenschutz. Also remove “Apple Sign In” references from AVV TOM Annex (“Zugangskontrolle”) and AGB §3(1). Optionally mark as “(geplant)” if launch is imminent.
File: landingpage/src/datenschutz.html:316–325
Critical Implement transactional emails (or adjust claims)
Legal claim: Datenschutz “E-Mail-Benachrichtigungen” lists 5 email types via Brevo: registration confirmation, gallery completion, cancellation confirmation, AGB/Datenschutz changes, offer changes. AGB §13 legally requires email notification of AGB changes.
Reality: mail.py is a stub — only notify_new_support_ticket() exists and it just logs, never sends. No Brevo SDK. No API key config. Zero emails are sent.
Fix: At minimum implement: (1) gallery-completion notification (core feature), (2) AGB/Datenschutz change notification (legally required by AGB §13). Full Brevo integration preferred. Alternatively, temporarily reduce Datenschutz claims to match reality during beta.
File: backend/app/services/mail.py
Critical Implement 30-day hard-delete job
Legal claim: AGB §6(5), Datenschutz “Registrierung”, AVV §10(1) all promise: “alle Daten innerhalb von 30 Tagen unwiderruflich gelöscht”.
Reality: Only soft-delete is implemented (deleted_at timestamp set, active=False). No scheduled job exists to hard-delete records or clean up GCS objects. Soft-deleted data persists indefinitely.
Fix: Create a management command / scheduled Cloud Run Job that: finds users/libraries/images with deleted_at older than 30 days, deletes associated GCS objects, then hard-deletes DB records. Run daily via Cloud Scheduler.
File: new backend/app/commands/cleanup.py or Flask CLI command
Critical Add AGB acceptance mechanism
Legal claim: AGB §3(1): “Der Vertrag kommt durch die Registrierung des Nutzers über Google Sign-In … und die Bestätigung dieser AGB zustande.”
Reality: Google Sign-In goes straight through to account creation. No AGB or Datenschutz acceptance checkbox or consent screen exists in the frontend. No acceptance timestamp stored in the DB.
Fix: After first successful Google OAuth, show an interstitial page with checkboxes for AGB + Datenschutz acceptance. Store agb_accepted_at and datenschutz_accepted_at timestamps in the User model. Block access until accepted.
Files: frontend/src/routes/login.tsx, backend/app/models.py

Part 2 — Medium: Should Fix Before Go-Live

These items are discrepancies that should be resolved but are less immediately risky than Part 1.
Medium Align server logfile claims with reality
Legal claim: Datenschutz “Hosting” states: “Server-Logfiles erstellt, die IP-Adresse, Zeitstempel, aufgerufene URLs, Browsertyp und Betriebssystem enthalten” with 30-day retention. AVV §2(2) lists “Protokolldaten der Clients: IP-Adresse, Zugriffszeit”.
Reality: No IPs are logged or stored by the application. _get_real_ip() extracts IP only for rate limiting (in-memory). No audit_logs table exists. Gunicorn may produce access logs to stdout, but these are not structured or retained with any defined policy.
Fix: Either (a) implement structured request logging that captures IP, URL, user-agent with 30-day retention in a log sink, or (b) update Datenschutz to reflect that only Cloudflare/GCP infrastructure logs exist and reference their retention policies.
Files: backend/app/main.py, landingpage/src/datenschutz.html:255–268
Medium Remove “Original als Vorschau” claim
Legal claim: Datenschutz “Sicherheitsmaßnahmen” and TOM Annex both state: “der Fotograf kann pro Bibliothek das Original als Vorschau aktivieren”.
Reality: No such feature exists. The Library model has no field for preview mode. All client previews are always watermarked.
Fix: Remove this claim from both Datenschutz and AVV TOM Annex. The current watermark-only behavior is more privacy-protective, so just describe reality.
Files: landingpage/src/datenschutz.html:132, landingpage/src/avv.html:249
Medium Implement data export capability
Legal claim: AVV §8(3): “Export von Kontodaten auf Anfrage an datenschutz@niecke-it.de”. AVV §10(2): “Vor der Löschung hat der Auftraggeber die Möglichkeit, seine Daten über die Plattformfunktionen herunterzuladen.” GDPR Art. 20 requires data portability.
Reality: No export endpoint. No download mechanism. No admin process documented.
Fix: Implement an API endpoint or admin action that packages user account data + photo originals for download. At minimum, define and document a manual admin process for DSAR requests.
File: new endpoint in backend/app/blueprints/api/
Medium Add cookie details table to Datenschutz
Legal claim: Datenschutz mentions “Session-Cookies zur Authentifizierung” generically.
Reality: The application sets a session cookie (Flask-Session, HttpOnly, Secure, SameSite=Lax). Frontend stores cookie_banner_dismissed in localStorage and lumios_token + lumios_google_profile in sessionStorage.
Fix: Add a table listing each cookie/storage item with name, purpose, duration, and type. Best practice seen in PicDrop and other German/Austrian providers. Also covers the localStorage items (which are technically not cookies but should be disclosed).
File: landingpage/src/datenschutz.html:271–282

Part 3 — Low: Track for Later

Non-blocking items to address over time or when related features are built.
Low Paddle payment integration (when leaving beta)
AGB §7, Datenschutz, AVV §7 reference Paddle.com as MoR. Not implemented yet. Acceptable during free beta but required before paid launch. Include Widerrufsrecht checkbox (AGB §8) in the subscription flow.
Low Fix CLAUDE.md: SendGrid → Brevo
CLAUDE.md references SendGrid as email provider. Legal docs and code TODOs reference Brevo. Internal inconsistency. Update CLAUDE.md to match.
File: CLAUDE.md
Low Disclose OpenTelemetry / Cloud Trace if used in production
OpenTelemetry with Cloud Trace or Jaeger OTLP exporter is implemented and configurable via CLOUD_TRACE_ENABLED. If enabled in production, traces may contain request metadata. Should be mentioned in Datenschutz under “Hosting” or as a separate section. Google Cloud would need to be referenced as sub-processor for this specifically.
File: backend/app/main.py:63–91
Low Consider “No AI Training” clause
Neither PicDrop nor Lumios explicitly address whether uploaded photos may be used for AI/ML training. Given current industry concerns, adding an explicit “Uploaded content will not be used for AI training” clause to AGB §6 and AVV §4 would be a differentiator and trust signal.
Low Add GDPR guide for photographers
PicDrop has a dedicated FAQ explaining GDPR implications for photographers (their role as data controller, when AVV is needed, portrait photo handling). Lumios has none. A simple help page would increase trust and reduce support load.
Low In-app AVV signing flow
PicDrop offers AVV signing directly in account settings under “Payments & Legal”. Lumios accepts AVV implicitly on registration. An explicit signing flow with downloadable PDF would be more transparent and is considered best practice in DACH markets.
